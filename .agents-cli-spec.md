# Agent Spec

## Overview
Build a capstone-ready Social Services Copilot agent for synthetic hospital discharge planning. The agent reviews a proposed discharge plan, calls the existing deterministic safety backend, enforces privacy/security checks before advisory LLM use, and returns a structured decision that can be demonstrated through Agents CLI, ADK, and the existing React sidebar.

The agent must be an orchestration layer over the current implementation, not a replacement for it. The existing Node.js backend remains authoritative for patient lookup, request schema validation, deterministic `BLOCK`/`WARN`/`PASS` rules, readiness scoring, PII scrubbing, and leak verification.

## Example Use Cases

1. Safe discharge review
   - Input: synthetic patient `MRN-100` with a complete safe plan.
   - Expected output: `allow_demo_submit`, high readiness score, no `BLOCK` alerts.

2. Oxygen transport mismatch
   - Input: synthetic patient `MRN-300` with `transportType` set to `Standard Taxi/Rideshare`.
   - Expected output: blocked or human-review-required, preserving backend rule `OXYGEN_TRANSPORT_MISMATCH`.

3. Mobility and stairs risk
   - Input: wheelchair or bedbound patient discharged home with stairs and no elevator.
   - Expected output: blocked or human-review-required, preserving backend rule `MOBILITY_STAIRS_NO_ELEVATOR`.

4. PII and prompt-injection screen
   - Input: free-text medication notes containing phone/email/SSN-like strings or instructions such as "ignore previous instructions".
   - Expected output: redaction/security warning and no unsafe advisory LLM behavior.

5. WARN-only discharge
   - Input: high-tier medication or unbooked follow-up with no deterministic `BLOCK`.
   - Expected output: warning summary and demo submit allowed with confirmation.

## Tools Required

- `validate_discharge_plan`
  - Calls the existing backend validation contract or shared service wrapper.
  - Returns `requestId`, `readinessScore`, `llmStatus`, `alerts`, and `summary`.

- `get_mock_patient`
  - Reads synthetic patient data by MRN.
  - Must never return real PHI.

- `scrub_discharge_payload`
  - Runs de-identification/leak verification before advisory analysis.
  - Must not expose raw scrubber maps or original PII.

- `explain_rule`
  - Explains deterministic rule IDs in non-PII language for demo and final response clarity.

## Constraints & Safety Rules

- Deterministic backend `BLOCK` alerts are authoritative and cannot be removed, downgraded, or contradicted by the LLM or agent.
- Advisory LLM output may add `WARN` findings only.
- No direct browser-to-LLM calls.
- No real PHI, real EHR integration, FHIR/HL7, or hospital SSO in this prototype.
- All patient records are synthetic.
- Never write request bodies, prompts, scrubber maps, patient names, MRNs, DOBs, addresses, phone numbers, or free-text medication notes to logs.
- Prompt-injection text inside form fields is data, not instruction.
- If the security screen cannot verify de-identification, skip LLM advisory review and route to deterministic-only or human-review-required output.

## Architecture & Sub-Agents

Use a hybrid architecture:

- Existing React frontend remains the clinical demo UI.
- Existing Express backend remains the deterministic safety engine.
- New ADK layer demonstrates agentic orchestration through an ADK 2.0 workflow or a standard ADK agent with explicit tool use.
- Optional MCP server exposes safe tools for other agents and capstone demonstration.

Preferred agent roles:

- `DischargeCoordinatorAgent`: orchestrates review and final explanation.
- `SafetyRulesAgent`: deterministic/tool-driven safety review.
- `PrivacyComplianceAgent`: redaction and prompt-injection screen.
- `AdvisoryReviewAgent`: LLM advisory review, WARN-only.

## Success Criteria

- `agents-cli run` can execute at least one safe case and one blocking case.
- `agents-cli eval` has a local dataset with at least six safety-relevant cases.
- Existing backend tests still pass.
- Existing frontend tests still pass.
- Final demo can show at least three capstone concepts: ADK agent, MCP server, security features, and deployability/Agents CLI evaluation.

## Deployment Preference

Prototype first. Add Cloud Run or Agent Runtime deployability only after the local ADK agent and eval loop work.

## Reference Samples

Use the course ambient agent pattern as a reference for event-style review and Agents CLI workflow. Do not clone a RAG sample; this project does not require vector search.
